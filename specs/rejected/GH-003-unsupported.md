# Rejected claims — GH-003: Follow favorite teams

**RETRACTED.** Nothing in the requirement is actually rejected — see
`specs/GH-003-plan.md` scenario 7 for the verified, passing behavior. This
file is kept (not deleted) as an audit trail of a false positive and its
root cause, per this framework's policy of never quietly erasing a finding.

## Original claim (now retracted below)

## "An anonymous (logged-out) visitor is prompted to log in when they try to
follow a team, rather than the follow silently failing or erroring." —
originally marked REJECTED_UNSUPPORTED

**What the first pass reported:** clicking "+ Follow" while logged out fired
`POST /api/teams/<teamId>/favorite`, which allegedly returned
`500 Internal Server Error`, followed by an uncaught client-side
`SyntaxError` trying to parse the empty error body — no login prompt, no
redirect, no user-visible error.

**Why this didn't hold up:** this plan was one of three (`GH-003`, `GH-004`,
`GH-005`) generated by Planner agents dispatched *concurrently* against the
same MatchDay dev server, sharing a single Playwright MCP browser session.
All three agents independently reported the same symptom during
exploration: auth state flipping between anonymous/fan/admin and
spontaneous navigation, with no action from that particular agent causing
it. Re-checking the actual source (`src/app/api/teams/[id]/favorite/route.ts`)
showed a `getSession()` guard that returns a clean `401` — not a 500 — for
an unauthenticated request, and a direct `curl` with no cookies confirmed
exactly that (`401 {"error":"Not authenticated"}`). The client-side
`FollowButton.tsx` also guards on a `loggedIn` prop and redirects to
`/login` *before* ever calling `fetch` when false — so a genuinely
anonymous click should never even reach the API.

**Re-verification:** performed directly (not via a Planner subagent, and
with no other browser-automation agent running concurrently) — logged out
via the UI, confirmed the nav showed "Log in"/"Sign up" immediately before
the click, then clicked "+ Follow". Result: clean client-side redirect to
`/login`, zero network requests to `/favorite`, no console errors beyond a
benign `401` on `GET /api/me`. This matches the requirement's acceptance
criterion exactly.

**Conclusion:** the original finding was contamination from a concurrent
agent's session (most likely another agent's login/logout mutating the
shared browser's cookies mid-interaction) landing on this agent's click at
the wrong instant, not a real defect in MatchDay. The actual product
behavior is correct. Retracted.

**Process takeaway, not a MatchDay bug:** never dispatch multiple Planner
(or Healer) agents concurrently — they share one MCP browser session. This
is now documented in `specs/GH-003-plan.md`'s "Notes for the Generator"
section and should be treated as a hard rule for running this framework.
